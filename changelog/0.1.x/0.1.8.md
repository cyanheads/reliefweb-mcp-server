# 0.1.8 — 2026-06-04

## Fixed

- **`reliefweb_get_report` / `reliefweb_get_disaster`** crashed on any valid ID because the ReliefWeb GET endpoint returns `{ data: [ {...} ] }` (array) but the service parsed it as `{ data: {...} }` (single object). Changed the type assertion to `RawRecord<T>[]` and unwraps with `data?.[0]`. ([#3](https://github.com/cyanheads/reliefweb-mcp-server/issues/3))
- **`reliefweb_get_country` / `reliefweb_get_disaster`** crashed with `flatMap is not a function` for any record with editorial profile data. The API returns `key_content`, `appeals_response_plans`, and `useful_links` as `{ title, active, archive }` objects, not flat arrays. Updated `RawCountryFields` and `RawDisasterFields` types; `normalizeCountryDetail` and `normalizeDisasterDetail` now concatenate `active` + `archive` before normalizing. ([#4](https://github.com/cyanheads/reliefweb-mcp-server/issues/4))
- **`reliefweb_list_sources`** — `types` field was always absent from results because `RawSourceFields.type` was typed as an array but the API returns a single object `{ name: string }`. Changed type to `{ name?: string }`; normalization now reads `f.type.name` directly. Tool description's `type` filter values corrected to exact API vocabulary (e.g. `"Non-governmental Organization"`, not `"NGO"`). ([#5](https://github.com/cyanheads/reliefweb-mcp-server/issues/5))
- **Service-level tests** added in `tests/services/reliefweb-service.test.ts` covering the GET array envelope, profile sub-field normalization, and source type single-object shape — 136 tests pass total.
